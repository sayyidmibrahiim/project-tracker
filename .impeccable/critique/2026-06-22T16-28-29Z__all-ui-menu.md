## Anti-Patterns Verdict

**Would someone believe "AI made this"?** Not at first glance. The auto-hiding dock, the risk-tiered confirmation gating (Draft vs Send, Preview vs Auto-send, reversible vs irreversible), and the Folder_State lock messaging are all genuinely considered — a real engineer thinking about banking-ops guardrails. **But three tells give it away on a second look:** the side-stripe accent border on cards/dialogs, the uncustomized Tailwind-default amber/indigo pill colors, and the Unicode-glyph nav icons. It reads as "a thoughtful functional shell whose visual vocabulary was never unified."

**LLM assessment:**

- **Side-stripe accent borders** on `.sc-card`, `.ru-card`, `.confirm-dialog`, and `.nav-btn[active]` — `border-left: 3px solid var(--color-dbs-red)`. This is an explicitly banned tell (accent side-stripe on cards/list items/alerts).
- **Coral is used for everything.** Primary CTA, danger/delete, active nav, state dropdowns, notification badges. When create, destroy, active-state, and "you have notifications" are the same hue, nothing stands out — the visual hierarchy collapses.
- **Uncustomized Tailwind palette leaks.** `#fef3c7`/`#92400e`/`#fde68a` (amber-100/800/200), `#eef2ff`/`#3730a3`/`#c7d2fe` (indigo-50/700/200) appear as literal pill colors in Scheduler/Rules. These are Tailwind defaults, not your DESIGN.md token system.
- **Unicode glyphs as the icon set.** Nav uses `⌂ ▧ ✦ ▥ ⚙ ◎`; DisabledHint uses `🔒 ⏳`; dismiss uses `×`; notifications use `●`. These render inconsistently across Windows font stacks and read as debug placeholders, not a designed icon system. `▧` for "Project Details" is unguessable.

**Deterministic scan (detect.mjs on 10 menu files):** **60 findings.**

- `design-system-color`: **42** — hardcoded literals outside DESIGN.md palette. Clustered in SchedulerActions (13) and RulesActions (12). Top offenders: `#111827`, `#DA1E28`, `#166534`, `#92400e`, `#fef3c7`, `#3730a3`, `#c7d2fe`.
- `design-system-radius`: **16** — `5px`, `7px`, `10px` outside the documented scale (4/6/8/12/16). On `.fa-btn`, `.oa-btn`, `.combo`, `.rm-menu`, `.header-year-pop`, `.confirm-btn`.
- `side-tab`: **2** — `border-left: 3px solid var(--color-dbs-red)` on `.sc-card` (SchedulerActions:320) and `.ru-card` (RulesActions:350). **The same pattern also lives in `ConfirmModal.svelte:129` and `styles.css:169/181` (`.nav-btn[active]`) but those files weren't in the scan set** — so the real count across the menu system is higher than the detector reported.

**False positives:** The `#DA1E28`/`#111827` literals in the `*Actions` button classes are _fallbacks_ inside `var(--color-dbs-red, #DA1E28)` — technically drift, but they'd only render if the token unsets. Still worth replacing with a token-only reference. The indigo/amber status pills are _not_ false positives; they're real uncaptured hues.

**Browser visualization:** Not run — this is a pywebview desktop app with no dev-server URL; the source review + detector cover the menu surfaces. No overlay to report.

## Overall Impression

The menu system is **functionally excellent and visually undisciplined.** The guardrails (Draft-before-Send, Folder_State locks, reversibility statements, recycle-bin deletes) are the work of someone who understands the domain. But the visual layer has no single source of truth: six button vocabularies, coral used as a universal accent, side-stripes and Tailwind defaults leaking in. The single biggest opportunity is **separating danger from brand** — give delete/send their own hue so the eye can find the risky thing instantly, which it currently cannot.

---

## What's Working

1. **Risk-tiered action design.** Draft (default, non-transmitting) vs Send (gated, irreversible-styled). Preview vs Auto-send (opt-in via setting). ConfirmModal states reversibility explicitly with a ↺ / ⚠ badge. This is genuinely good operations UX — the kind of thinking that prevents a support ticket.
2. **The auto-hiding dock, mechanically.** Frees vertical space for a dense dashboard, reappears on idle, stays open on `:focus-within` and `:has(.open)`, has `prefers-reduced-motion` handling. As an _interaction_ it's well-engineered — the problem is discoverability, not the mechanism.
3. **Honest Folder_State locks.** `DisabledHint` names the exact state and lock reason; the backend is authoritative; the control is genuinely non-activatable. No fake-disabled-then-actually-works surprises.

## Priority Issues

### [P1] Danger actions share the brand color

- **Why it matters:** `.rm-danger`, `.fa-btn-danger`, `.oa-btn-danger`, `.ta-btn-danger`, `.sc-btn-danger`, `.ru-btn-danger` all use `--color-dbs-red` → `--primary` (`#cc785c` coral). Delete, Send email, Send Teams, and "Add Project" are the same hue. A user scanning a row cannot visually distinguish "create" from "destroy." The app already defines `--error: #c64545` and tag-red tokens (`--tag-red-bg/ink`) that are unused by danger states. This is a real error-prevention failure on a banking-ops tool.
- **Fix:** Route all danger/destructive states through a dedicated `--danger` token (`--error` or a new `--danger`/`--danger-hover` pair distinct from `--primary`). Update the six `*-btn-danger` classes + `.rm-danger` + the irreversible ConfirmModal proceed button. Keep coral for primary CTAs and active nav only.
- **Suggested command:** `/impeccable colorize` (establish the danger role), then `/impeccable polish`.

### [P1] Side-stripe accent border across cards and dialogs (banned tell)

- **Why it matters:** `border-left: 3px solid var(--color-dbs-red)` on `.sc-card`, `.ru-card`, `.confirm-dialog`, and `.nav-btn[active]`. The skill bans `border-left/right > 1px as a colored accent on cards, list items, callouts, or alerts` — it's the most identifiable "AI made this" tell present here.
- **Fix:** Remove the side-stripe. For cards, use a full 1px hairline border + a leading icon or a small leading number/tag. For `.nav-btn[active]`, signal active state with background fill + a 3px **bottom** indicator (or a leading dot), not a left rail. For ConfirmModal, lead with the ⚠ icon (already present) and drop the stripe.
- **Suggested command:** `/impeccable quieter` (strip the tell) → `/impeccable polish`.

### [P1] Six parallel button vocabularies in the \*Actions family

- **Why it matters:** `.fa-btn`, `.oa-btn`, `.pa-btn`, `.ta-btn`, `.sc-btn`, `.ru-btn` each redefine a near-identical button with slightly different padding/radius/hover. `styles.css` already has `.btn-primary/.btn-secondary/.btn-tiny/.btn-danger/.btn-black`. Result: drift (5px radius in some, 7px in others), inconsistency, and 6 places to change when the button style evolves. The detector's 16 radius hits and most color hits cluster here.
- **Fix:** Collapse the six into the shared button classes (or one new `.btn-ghost` if needed for the small inline action style). Each `*Actions` component should use `.btn-secondary` / `.btn-danger` / `.btn-tiny`, not its own class. Remove the `#DA1E28`/`#111827` fallback literals.
- **Suggested command:** `/impeccable distill` (unify the controls) → `/impeccable polish`.

### [P2] Dock auto-hide punishes the active working style the product is built for

- **Why it matters:** The primary navigation disappears on any `mousemove`/`keydown` and only returns after 750–1200 ms idle. The product's persona is a DBS engineer _actively_ working many projects — exactly the user who will never see their own nav. Labels are hidden until hover, and the dock fades to opacity .14 (nearly invisible) when hidden. A first-timer (Jordan) will conclude there is no navigation.
- **Fix:** Either (a) make the dock persistent with a manual toggle, (b) raise the hidden opacity so a sliver is always visible as an affordance, or (c) only auto-hide after a longer true-idle timeout (e.g. 3–5 s) and reveal on any edge-cursor approach, not on every mousemove. Always show labels on the active item.
- **Suggested command:** `/impeccable polish` (interaction tuning).

### [P2] Header is overstuffed; decorative datetime occupies premium center real estate

- **Why it matters:** The 58px header crams page title, a live-updating datetime badge, year `<select>`, ＋ add-year, add-year popover, search, Add Project, and refresh. The datetime badge sits in the center column updating every second — purely decorative on a desktop app where the OS taskbar already shows a clock. The add-year popover overlaps the search input when open. Cognitive load: ~7 controls in one bar plus the hidden dock.
- **Fix:** Drop the datetime badge (or move it to a status strip). Group year + add-year into a single "Year" dropdown control. Keep header to: title | search | primary action. Move refresh into the row/table toolbar where the data actually lives.
- **Suggested command:** `/impeccable layout` → `/impeccable distill`.

### [P2] DashboardRowMenu lacks keyboard navigation and focus management

- **Why it matters:** The overflow menu has `role="menu"`/`role="menuitem"` and Escape-to-close (good), but: focus is not moved into the menu on open, there is no arrow-key navigation between items, no `aria-activedescendant`, and the menu closes on scrim click but not on `focusout`/`blur`. Keyboard-only users (Sam) and power users (Alex) can't operate it without the mouse.
- **Fix:** On open, move focus to the first menuitem. Implement Up/Down arrow cycling, Home/End, and Enter/Space activation. Close on `focusout` when focus leaves the menu root. Add `aria-activedescendant` synced to the active item, or use roving `tabindex`.
- **Suggested command:** `/impeccable harden` (a11y) → `/impeccable audit`.

### [P2] Status pills introduce untokenized Tailwind-default hues

- **Why it matters:** `.sc-pill-info` (indigo `#eef2ff`/`#3730a3`/`#c7d2fe`), `.sc-pill-warn`/`.ru-pill-warn` (amber `#fef3c7`/`#92400e`/`#fde68a`), and `.ta-ok`/`.fa-ok` (`#166534`) are literal Tailwind palette colors with no DESIGN.md token. They introduce three new hue families (indigo, amber, green) that fight the warm coral/cream system. The detector's 42 color findings are mostly these.
- **Fix:** Map info/warn/success to the existing semantic tokens: `--tag-amber-*` (already defined) for warn, create `--tag-info-*` if indigo is truly wanted (justify it), and route `.ta-ok`/`.fa-ok` through a `--success-ink` token. Remove the literals.
- **Suggested command:** `/impeccable colorize` → `/impeccable extract` (pull them into the system).

### [P3] Unicode-glyph nav icons are unguessable and font-fragile

- **Why it matters:** `⌂ ▧ ✦ ▥ ⚙ ◎` for the six nav destinations. No user infers "Project Details" from `▧`. Glyph rendering varies across Windows font fallbacks (some show tofu boxes). The dock is icon-only by default, so this is the _only_ label until hover.
- **Fix:** Replace with a small inline-SVG icon set (the codebase already uses inline SVGs in Header for search/refresh/add — follow that precedent). Keep the hover label; consider always-visible labels for the active item.
- **Suggested command:** `/impeccable delight` (icon set) or `/impeccable polish`.

### [P3] `dash-state-select` is a solid coral dropdown — too loud, and reused per row

- **Why it matters:** Every CR/Drone state picker is a filled coral `<select>` styled like a primary CTA. A dashboard row can show 2+ of them, plus the row overflow button — a lot of coral in one row, indistinguishable from "Add Project." State is data, not an action; it shouldn't look like a button.
- **Fix:** Style state picks as a subtle outlined control (hairline border, white/canvas fill, ink text) with a small colored status dot reflecting the state. Reserve coral for the active/selected state only.
- **Suggested command:** `/impeccable quieter` → `/impeccable polish`.

### [P3] Radius drift (5/7/10 px) outside the documented scale

- **Why it matters:** DESIGN.md documents 4/6/8/12/16/9999. The menu buttons and popovers use 5 px (`.fa-btn`), 7 px (`.combo`, `.dash-state-select`), 10 px (`.rm-menu`, `.header-year-pop`). Minor, but it's the "slightly off" feeling that accumulates.
- **Fix:** Snap every menu-surface radius to the nearest token: 5→4 or 6, 7→6 or 8, 10→8 or 12. Mostly resolved for free by the P1 button-unification work.
- **Suggested command:** resolved via `/impeccable distill` + `/impeccable polish`.

- [✓] **Single focus** — FAIL on dashboard: header (7 controls) + dock + 2 state selects per row + row action.
- [~] **Chunking** — Automations tabs chunk well; header doesn't.
- [✓] **Grouping** — PASS within panels.
- [✗] **Visual hierarchy** — FAIL: coral = primary + danger + active-nav + state + notification. Everything competes; nothing wins.
- [✓] **One thing at a time** — PASS.
- [~] **Minimal choices** — BORDERLINE: Scheduler/Rules card action rows show 4–6 ungrouped buttons (Evaluate / Edit / Disable / Logs / Delete).
- [✓] **Working memory** — PASS (state stays co-located with the item).
- [~] **Progressive disclosure** — PARTIAL: Scheduler/Rules forms and add-year popover reveal inline (good); header shows everything always.

**Failed items: 2 (hierarchy, single-focus). Moderate cognitive load — address the hierarchy collapse (P1 danger-as-primary) and it cascades.**

## Persona Red Flags

**Alex (Power User / Dashboard + row menus):**

- No keyboard navigation between dashboard rows; no bulk-select or batch state-change across projects.
- Row overflow menu (⋯) is mouse-only — no arrow keys, no shortcut to open.
- The dock vanishes the moment he starts actively working (mousemove hides it). The single most efficient user is the most punished by the dock.
- Repetitive one-at-a-time CR-state changes across 20 projects with no accelerator.

**Jordan (First-Timer / first dashboard load):**

- Icon-only dock with opaque glyphs (`▧` = Project Details is unguessable); labels hidden until hover.
- Dock auto-hides on first mouse move — Jordan will conclude "there is no navigation" and never find Settings/Report.
- Template-category jargon (`ACK_UAT`, `APRVL_SOP`) in Outlook/Teams actions is unexplained; no inline help.
- Add-Year is a ＋ button inside the year cluster — not obviously a separate action.

**Sam (Keyboard / Screen-reader / menus + modals):**

- `DashboardRowMenu` opens without moving focus into the menu; no arrow-key item cycling; no `aria-activedescendant`.
- Dock nav buttons have `aria-label` (good) and `:focus-within` keeps the dock open (good).
- State picks are native `<select>` (good); ConfirmModal is `role="alertdialog"` with labelledby/describedby (good).
- Unicode nav glyphs may be read as decorative or as the literal character by some SR configs — the `aria-label` saves it, but an SVG set would be cleaner.
- Missing: focus trap in ConfirmModal (focus can leave the dialog via Tab).

**Project-specific — "Ibrahim" the DBS Ops Engineer (from PRODUCT.md):**

- Manages many CR projects under time pressure; needs state-at-a-glance and fast action. Coral-everywhere means he must _read_ every button before clicking — he can't rely on color to spot the destructive one. That's friction on every row, every session.
- Dock auto-hide fights his active working style.
- No bulk actions = repetitive single-row edits when 10 projects move to UAT on the same day.

## Minor Observations

- `.refresh-button` spin is a fixed 650 ms `setTimeout`, not tied to actual fetch completion — can stop spinning before data arrives, or spin forever if the fetch hangs.
- ConfirmModal has no focus trap; Tab escapes the dialog. Also no `autofocus` on Cancel (the safer choice) — focus stays on the triggering control behind the overlay.
- `dock-hover-zone` is 720 px wide × 84 px tall at the screen bottom — a large invisible click target that sits over dashboard content; clicks in that band hit the zone, not the table.
- `.dock-badge` unread count is coral-on-white — same as primary CTAs; a notification count usually warrants a distinct (e.g. error/red) treatment.
- Sidebar `formatTime` and Header `nowParts` both exist; the dock uses en-GB dates, the header uses en-US — inconsistent locale formatting in one app.
- `Automations.svelte` Reminder tab hardcodes four `metric-card`s with emoji icons (⌛ ! Ⅱ 🔔) — placeholder feel.

## Questions to Consider

- If danger had its own hue, how much faster would Ibrahim scan a 20-row dashboard for the one project he must not delete?
- The dock hides to free space — but the dashboard already scrolls horizontally (min-width 1540 px). Is vertical space actually the constraint, or is the dock solving a problem that isn't there?
- Template codes (ACK_UAT, APRVL_CR…) are meaningful to a DBS engineer but cryptic otherwise. Should the selector show the human label ("Acknowledge UAT") with the code as a secondary detail, instead of the raw code?
- Six button classes, three danger treatments, two icon systems. What would a single disciplined pass buy you in consistency _and_ maintenance?
- The datetime badge updates every second in the center of the header. What is the user doing with that information that the Windows taskbar doesn't already provide?
